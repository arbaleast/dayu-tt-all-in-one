| Phase | 名称 | 设计完成度 | 实际落地度 | 当前状态 |
|-------|------|-----------|-----------|---------|

## 二、Phase 0 Obico 集成效果评估

### 2.1 已实现的部分

- Obico 账号体系（app.obico.io）已配置
- Moonraker API 授权已完成
- Telegram Bot 推送通道已建立（@BotFather Token）
- 摄像头物理架设已完成（USB 摄像头）
- Moonraker-Obico 插件已集成到 Klipper

### 2.2 缺失的验证步骤

设计文档明确要求 Phase 0 完成后必须验证三项：
1. App 实时画面正常
2. **制造一次失败，观察 Telegram 3 分钟内收到通知** ← **未做**
3. App 可远程暂停/取消打印

### 2.3 实际问题推测

根据上下文判断，Obico 目前可能存在以下问题之一或多个：
- **免费额度耗尽风险**：Obico 免费层每月仅 10 AI 小时，若打印任务多会快速用完
- **检测灵敏度未调优**：默认阈值可能产生误报或漏报
- **Moonraker 连接稳定性**：未做长时间稳定性测试
- **夜间/低光环境失效**：未配置补光灯，夜间检测率下降

## 三、调研报告技术选型对比

### 3.1 设计文档 vs 调研报告差距

| 技术方向 | 设计文档选型 | 调研报告补充 | 差距 |
|---------|------------|------------|------|
| Input Shaping 校准 | 未详细说明 | **Klippain Shaketune + ADXL345**（生产级，¥15）| **重大缺口**：大鱼 TT 尚未用 ADXL345 做 IS 校准，这是 3D 打印 AI 校准最成熟、最有效的第一步 |
| PA 校准 | 传统方法（测试塔） | **PressureAdvanceCamera**（分钟级，CV 自动测量，¥30 USB 内窥镜）| **重大缺口**：PA 是影响打印质量核心参数，设计文档未跟进调研报告的自动化方案 |
| 视觉故障检测 | Obico（生产级 ✓） | Obico / OctoEverywhere / OctoPrint-PiNozCam | Obico 选型正确，但缺少备选方案 |
| 本地 VLM 质检 | 未规划 | **LLM-3D Print 多 Agent 框架**（Monitor→Reasoner→Executor）| **前瞻性缺口**：调研报告指出 GPT-4o 等 VLM 已具备 in-context learning，可直接用于缺陷检测，无需额外训练 |
| RAG 知识管理 | 未规划 | **LangChain/LlamaIndex + Chroma/Qdrant + Ollama** | **重大空白**：大鱼 TT 有大量 Klipper 配置经验、故障排查记录，适合用 RAG 建立知识库 |
| 自演化控制 | 规则+ML 混合引擎 | **Reinforcement Learning 闭环**（ResearchGate 2026）| 调研报告有 RL 前沿探索，但距离落地较远，设计文档方向合理 |
| 高速改装 | 未涉及 | **AWD 四驱改装**（LDO 套件 ¥350，速度 600mm/s+）| 大鱼 TT 是否需要 AWD 取决于用途，缺需求定义 |

### 3.2 调研报告可直接采纳的快速收益项

以下三项成本低、效果立竿见影，**建议立即补充进 Phase 0-1**：

1. **ADXL345（¥15）+ Klippain Shaketune**：消除 80-95% ringing，加速度上限翻倍，是目前性价比最高的单项升级
2. **PressureAdvanceCamera（¥30 USB 内窥镜）**：PA 校准从 10-30 分钟压缩到分钟级，精度更高
3. **BME280（¥20）腔体温湿度监测**：为 Phase 1-2 的进化模型提供环境特征，是进化学习的重要输入

## 四、具体可落地改进建议

### 4.1 立即行动（本周，零阻力）

#### 建议 1：补全 Phase 0 末端验证

**问题：** Phase 0 声称完成，但从未真正制造失败来验证 Obico 检测链路。

**动作：**
```bash
# 1. 人为制造一次已知失败来测试全链路
# 方式A：打印开始后手动把平台降低 5mm（触发粘连检测）
# 方式B：故意不加载耗材（触发缺料检测）
# 方式C：打印前把热端温度降低 30°C（触发温度异常）

# 2. 记录 Telegram 推送延迟是否 < 3 分钟
# 3. 验证 Moonraker 远程暂停/取消是否生效
```

**交付：** 填写 AI校准进化实战指南.md Phase 0 验证清单 ✅

#### 建议 2：采购 ADXL345（¥15），启动 Input Shaping 校准

**问题：** 大鱼 TT 目前使用出厂默认加速度参数，未做机器特定校准，是最大的效率/质量损失点。

**动作：**
```bash
# 树莓派上检查 ADXL345 是否已连接
sudo i2cdetect -y 1
# 如果显示 53 地址说明 ADXL345 已识别

# Klippain Shaketune 安装
wget -O ~/klippain.sh https://raw.githubusercontent.com/Frix-x/klippain/main/install.sh
bash ~/klippain.sh
```

**预期收益：** ringing 消除 80-95%，可安全提升加速度到 8,000-13,000 mm/s²

### 4.2 短期（1-2 周）

#### 建议 3：Phase 1 优先实现数据回流（最小可行闭环）

**问题：** Phase 1 原设计包含 BME280 + Flask 面板 + MQTT 等多项并行任务，过于复杂导致推进缓慢。

**改进思路：** 简化 Phase 1，用最小代码先跑通"打印完成 → 数据写入 SQLite"这条链路。

**最小可行 Phase 1：**
```python
# 1. 在 NAS 上创建 SQLite 数据库（3张表）
# print_jobs, param_adjustments, param_knowledge

# 2. 写一个 Moonraker webhook 脚本
# 每次打印完成/失败 → 自动查询本次打印元数据 → 写入 SQLite
# 不需要 BME280，不需要 MQTT，直接用 Moonraker 的 notify_status

# 3. 写一个简单的 /print_history 命令
# Telegram 查询最近 N 次打印记录
```

**交付：** Telegram 能查询打印历史，SQLite 有真实数据

#### 建议 4：采购 PressureAdvanceCamera 套件（¥30）

**问题：** PA 是大鱼 TT 当前最缺乏自动校准的核心参数，manual 测试塔耗时长且精度低。

**动作：**
- 购入 USB 内窥镜（约 ¥30）
- 安装 PressureAdvanceCamera（[GitHub](https://github.com/)
- 运行一次自动校准，将结果写入 Klipper 配置

### 4.3 中期（3-6 周）

#### 建议 5：Phase 2 从"特征工程"转向"RAG 知识库"

**问题：** 设计文档 Phase 2 的 scikit-learn 参数预测引擎，需要大量训练数据才能生效（需要 Phase 1 先跑几周才有足够样本）。

**改进：** 在等待 Phase 1 数据积累期间，先做 RAG 知识库，这个不依赖打印数据。

**RAG 优先做：**
```
docs/
├── Klipper 配置经验（已沉淀）
├── 故障排查手册（已有内容）
├── 材料参数矩阵（PLA/PETG/ABS/TPU 基础值）
└── 大鱼 TT 特异性调参记录

↓
用 LlamaIndex + Ollama Qwen2.5-7B 建立本地知识库
↓
"上次这个模型打印失败是什么原因？" → AI 直接回答
"这个材料用什么参数最好？" → AI 推理回答
```

**收益：** 立即可用，不用等数据积累，是大鱼 TT 当前最差异化的能力

#### 建议 6：Phase 3 MPU6050 降级为 ADXL345（已有）

**问题：** Phase 3 设计 MPU6050 用于振动检测，但 ADXL345 实际上已经在手（用于 IS 校准），且 ADXL345 采样率更高（比 MPU6050 更适合振动分析）。

**改进：** 用同一个 ADXL345 同时服务：
- Input Shaping 校准（Phase 0/1）
- 振动异常检测（Phase 3，FFT 主频分析）

不需要额外购买 MPU6050，复用 ADXL345 即可。

### 4.4 架构层改进

#### 建议 7：将"AI 校准进化"拆分为两个独立并行 track

**当前问题：** 设计文档把 AI 进化做成线性 Phase 依赖链，但实践中依赖链导致全面拖延。

**改进方案——两个 track 并行：**

```
Track A：硬件校准自动化（不依赖进化数据）
  Week 1: ADXL345 IS 校准（立竿见影）
  Week 2: PressureAdvanceCamera PA 校准
  Week 3: BME280 腔体监控

Track B：数据与知识系统（不依赖硬件）
  Week 1: SQLite 数据管道（Moonraker webhook）
  Week 2: RAG 知识库 + Telegram AI 助手
  Week 3: 打印历史分析报告

Week 4+: 两个 track 合并 → Phase 3 故障恢复闭环
```

#### 建议 8：在 Phase 0 补充"Obico 免费额度管理"策略

**问题：** Obico 免费层每月 10 AI 小时，容易在密集打印时耗尽，导致关键时刻 AI 检测失效。

**建议策略：**
- 估算大鱼 TT 月均打印时长，合理规划 Obico 使用
- 若月均打印 > 10 小时：考虑 OctoEverywhere（免费无限量）作为备份
- 或部署 3DPrintSentinel（完全本地，无 AI 小时限制）

## 五、总结：优先级排序

| 优先级 | 行动 | 成本 | 预期收益 | 时间 |
|-------|------|------|---------|------|
| **P0** | Phase 0 末端验证（制造失败测试） | ¥0 | 确认链路有效 | 30 分钟 |
| **P0** | ADXL345 IS 校准 | ¥15 | 速度翻倍，质量提升 | 1-2 小时 |
| **P1** | SQLite 数据管道（Moonraker webhook） | ¥0 | 进化数据底座 | 1 天 |
| **P1** | PressureAdvanceCamera PA 校准 | ¥30 | 精度提升，时间节省 | 30 分钟 |
| **P2** | RAG 知识库 + 本地 AI 助手 | ¥0 | 差异化能力立竿见影 | 1 周 |
| **P2** | BME280 腔体监控 | ¥20 | 环境特征采集 | 1-2 小时 |
| **P3** | Phase 3 双传感器故障恢复 | ADXL345 已覆盖 | 无人值守打印 | 2-3 周 |

**核心结论：** 大鱼 TT 当前最大的问题不是设计不足，而是**设计过度、落地不足**。Phase 0-5 的文档架构已经足够完善，但 Phase 1-5 的实际执行进度几乎为零。建议将 Phase 2 的 scikit-learn 参数预测推后，优先搭建 RAG 知识库和 SQLite 数据管道，这两个不需要等待硬件改造，且能立即产生用户可感知价值。
